What Microsoft does well

2022-02-04

The three hardest problems in software engineering is off by one errors and naming.

As someone who has used Microsoft tooling for less than 6 months, I have been pleasantly surprised at how good Microsoft's naming conventions are.

This is a collection of examples I came up with:

The C# Language

The convention for C# classes and methods are to be descriptive (this should be true for all languages). It took me a while to get used to what I confused descriptive naming for verbosity. This may be as a result of working on better written software project.

Exploring the Azure .NET SDK code base (specifically the keyvaultexamples) was made easier due to naming conventions. Within the code, complexity of the domain was pushed to lower level components meaning I could follow the non-domain specific logic.

Azure

Azure offers a lot of services. Starting out, I was overwhelmed with what each service did and wondered if I would ever remember which service did what. After spending some time with the Microsoft Learn resources and Pluralsight courses, I felt as if given a new resource, I could infer what it's purpose is. Compared to AWS, Azure services are well named and easier to remember.

For example, which is easier to remember AWS S3 (Amazon Simple Storage Service) or Azure Storage. For compute, AWS EC2 (Amazon Elastic Compute Cloud) or Azure Compute. For hosting apps, AWS Elastic Beanstalk or Azure App Service. For DNS, AWS Route 53 or Azure DNS.

If I worked with AWS, maybe I would have become as familiar with the resource names just as quick. From an outsiders perspective or new-comer, the overhead of trying to remember clever names doesn't substitute for clarity.

PowerShell

Despite not having the GNU Core utilities such as less and grep on a fresh install there are some usable alternatives. For grep, there is Select-String. Listing files with ls is still possible, though the optional flags are different. To show hidden files you have to use the -Hidden flag. Turns out ls is an alias for Get-ChildItem along with the command prompts dir.
